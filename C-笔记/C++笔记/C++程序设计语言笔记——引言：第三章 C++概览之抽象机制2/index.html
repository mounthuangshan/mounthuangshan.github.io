0 抽象类型

complex 和 Vector 等类型之所以被称为具体类型（concrete type），是因为它们的表现形式属于定义的一部分。在这一点上，它们与内置类型很相似。相反，抽象类型（abstract type）则将使用者与类的实现细节完全隔离开来。为了做到这一点，我们分离接口与表现形式并且放弃了纯局部变量。因为我们对抽象类型的表现形式一无所知（甚至对它的大小也不了解），所以必须从自由存储为对象分配空间，然后通过引用或指针访问对象。

下面是对上面的总结：

  • 具体类型（Concrete Type）
    1.定义：具体类型是指其表现形式（即内存布局和成员数据）是定义的一部分的类型。例如，complex 和 Vector 类型就是具体类型，因为它们的实现细节（如内存布局）是已知的。
    2.特性：与内置类型相似，具体类型的实例化对象可以直接使用。
  • 抽象类型（Abstract Type）
    1.定义：抽象类型将使用者与类的实现细节完全隔离开来。用户只能通过接口与抽象类型交互，而无法直接访问其内部实现。
    2.实现：通过定义包含纯虚函数（pure virtual function）的类来实现抽象类型。纯虚函数用 = 0 标记，表示派生类必须提供该函数的实现。
    3.内存管理：由于抽象类型的具体表现形式未知，通常使用动态内存分配（通过引用或指针）来管理抽象类型的对象。
  • 多态类型（Polymorphic Type）
    1.定义：多态类型是指可以通过基类接口访问不同派生类实现的类型。
    2.实现：通过继承关系和虚函数实现多态性。基类定义接口，派生类提供具体实现。
    3.用途：多态性允许在不知道具体对象类型的情况下调用对象的方法，从而提高了代码的灵活性和可扩展性。
  • 继承（Inheritance）
    1.定义：继承是面向对象编程中的一种机制，允许一个类（派生类）继承另一个类（基类）的属性和方法。
    2.作用：继承实现了代码的重用和扩展。通过继承，派生类可以获取基类的功能，并在此基础上添加新的功能或修改现有功能。
  • 代码分析
    1.Container 类：作为抽象基类，定义了 operator[] 和 size() 两个纯虚函数，以及一个虚析构函数。这个类提供了一个接口，派生类必须实现这些接口。
    2.Vector_container 类：派生自 Container 类，使用 Vector 类型作为内部存储，并实现了 Container 类中的纯虚函数。
    3.List_container 类：另一个派生自 Container 类的类，使用 std::initializer_list 作为内部存储，并实现了 Container 类中的纯虚函数。这个类展示了如何使用不同的内部数据结构来实现相同的接口。
    4.use 函数：一个接受 Container 引用作为参数的函数，展示了多态性的使用。它不知道传递给它的具体对象类型，但可以通过 Container 接口调用对象的方法。
  • 关键点
    1.接口与实现分离：抽象类提供了一种定义接口的方式，而派生类则负责实现这些接口。这种分离使得代码更加模块化和易于维护。
    2.动态多态性：通过基类指针或引用来调用派生类的方法，实现了动态多态性。这允许在运行时决定使用哪个派生类的实现。
    3.灵活性：多态性提高了代码的灵活性，因为可以在不修改现有代码的情况下添加新的派生类。
    内存管理：由于抽象类型的具体实现未知，通常需要使用动态内存分配来管理对象。这增加了内存管理的复杂性，但提供了更大的灵活性。
  • 注意事项
    1.虚析构函数：对于包含虚函数的类，应该定义一个虚析构函数。这确保了当通过基类指针删除派生类对象时，能够正确调用派生类的析构函数。
    2.性能考虑：虽然多态性提供了很大的灵活性，但它也增加了函数调用的开销（因为需要查找虚函数表）。在性能敏感的应用中，需要权衡这些开销。
    这里你可能有一个问题，C++是如何判断什么时候用重载运算符，什么时候用内置运算符的，规则如下：
  • 函数名和语法形式：
    重载的运算符是通过成员函数或友元函数来实现的，这些函数具有特殊的函数名，即operator后跟运算符符号。例如，重载加法运算符的函数名是operator+。

当在表达式中使用+运算符，并且操作数的类型是MyClass或MyClass的派生类时，编译器会查找这个重载的operator+函数。

  • 作用域和可见性：
    重载的运算符函数必须在其作用域内可见。如果运算符函数是类的成员函数，则它自然在类的作用域内。如果运算符函数是友元函数，则它必须在友元声明中提及，并且在调用时可见。
  • 参数匹配：
    编译器会根据重载运算符函数的参数列表来确定哪个函数应该被调用。对于成员函数，左操作数（通常是*this）是隐式的，而右操作数（或额外的操作数，对于多元运算符）是显式的函数参数。
  • 内置运算符的优先级和结合性：
    内置运算符具有固定的优先级和结合性，这些规则在C++语法中是硬编码的。重载的运算符不会改变这些规则；它们只是提供了在特定类型上应用这些规则的新方式。
  • 类型转换：
    如果重载的运算符函数的参数类型与操作数的类型不完全匹配，编译器会尝试应用类型转换规则来找到匹配的函数。这包括用户定义的转换函数和编译器生成的转换（如从派生类到基类的转换）。
  • 错误处理和歧义消除：
    如果存在多个可能的重载运算符函数（例如，通过继承获得的多个重载），编译器会应用重载解析规则来确定哪个函数应该被调用。如果无法确定唯一的函数，编译器将报错，指出重载冲突或歧义。

当我们通过派生的方式定义新类时，可以向其中添加数据成员或者新的操作，或者都添加。这种机制一方面提供了巨大的灵活性，同时也可能带来混淆，从而造成糟糕的设计。总的来说，类层次提供了两种便利：

  • 接口继承（Interface inheritance）：派生类对象可以用在任何需要基类对象的地方。也就是说，基类看起来像是派生类的接口一样。Container 和 Shape 就是很好的例子，这样的类通常是抽象类。
  • 实现继承（Implementation inheritance）：基类负责提供可以简化派生类实现的函数或数据。Smiley 使用 Circle 的构造函数和 Circle：draw（）就是例子，这样的基类通常含有数据成员和构造函数。

这里需要扩展一下C++的智能指针：
C++11 引入了 3 个智能指针类型：

  • std::unique_ptr ：独占资源所有权的指针。
  • std::shared_ptr ：共享资源所有权的指针。
  • std::weak_ptr ：共享资源的观察者，需要和 std::shared_ptr 一起使用，不影响资源的生命周期。
  • std::auto_ptr 已被废弃。

下面是介绍：

  • std::unique_ptr
    简单说，当我们独占资源的所有权的时候，可以使用 std::unique_ptr 对资源进行管理——离开 unique_ptr 对象的作用域时，会自动释放资源。这是很基本的 RAII 思想。
  • std::shared_ptr
    std::shared_ptr 其实就是对资源做引用计数——当引用计数为 0 的时候，自动释放资源。一个 shared_ptr 对象的内存开销要比裸指针和无自定义 deleter 的 unique_ptr 对象略大。
    无自定义 deleter 的 unique_ptr 只需要将裸指针用 RAII 的手法封装好就行，无需保存其它信息，所以它的开销和裸指针是一样的。如果有自定义 deleter，还需要保存 deleter 的信息。
    shared_ptr 需要维护的信息有两部分：
    1.指向共享资源的指针。
    2.引用计数等共享资源的控制信息——实现上是维护一个指向控制信息的指针。
  • std::weak_ptr
    std::weak_ptr 要与 std::shared_ptr 一起使用。 一个 std::weak_ptr 对象看做是 std::shared_ptr 对象管理的资源的观察者，它不影响共享资源的生命周期：
    如果需要使用 weak_ptr 正在观察的资源，可以将 weak_ptr 提升为 shared_ptr。
    当 shared_ptr 管理的资源被释放时，weak_ptr 会自动变成 nullptr。
    当 shared_ptr 析构并释放共享资源的时候，只要 weak_ptr 对象还存在，控制块就会保留，weak_ptr 可以通过控制块观察到对象是否存活。
