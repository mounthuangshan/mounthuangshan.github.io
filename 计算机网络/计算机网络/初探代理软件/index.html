V2Ray 的架构由多个模块组成，每个模块负责特定的功能，确保系统的灵活性和可扩展性。

  • 核心（Core）：负责处理网络流量的转发、加密和解密等基本功能。
  • 协议支持：V2Ray 支持多种代理协议，如 VMess、VLess、Socks、HTTP、Shadowsocks 等。

  • 路由（Routing）：允许用户根据不同的规则将流量导向不同的代理服务器，实现灵活的流量管理。

在开始之前需要介绍一下系统代理和 VPN 的区别：

系统代理是一种网络配置，允许特定的应用程序通过指定的代理服务器发送和接收网络请求。代理服务器充当客户端和目标服务器之间的中介。当配置了系统代理后，支持代理设置的应用程序（如浏览器）会将其网络请求发送到代理服务器，代理服务器再将请求转发到目标网站或服务，并将响应返回给应用程序。系统代理本身不一定提供加密功能，取决于代理协议（如HTTPS代理可以加密流量，SOCKS5代理通常不加密）。系统代理通常工作在应用层（OSI模型的第七层），即它只代理特定应用程序的网络流量，如Web浏览器、邮件客户端等，并且系统代理仅对支持代理设置的应用程序有效，其他不支持代理的应用程序的流量不会通过代理服务器。系统代理相当于启用了一个 TCP 应用，侦听特定的端口等待应用程序连接使用。应用程序需要通过系统代理访问网络，会首先连接至代理客户端，然后由代理客户端转发网络请求，达到科学上网的目的。
VPN是一种通过加密隧道将整个设备的网络流量转发到远程服务器的技术，旨在保护数据隐私、绕过地理限制并提高网络安全性。VPN客户端软件在用户设备和VPN服务器之间创建一个加密隧道，所有通过设备的网络流量（包括系统和应用程序级别的流量）都会被加密并转发到VPN服务器，然后再由服务器将流量发送到目标地址。VPN工作在网络层（OSI模型的第三层），即它拦截并转发设备的所有网络流量，无论是应用程序还是系统级别的请求，一旦连接成功，设备上的所有网络流量（包括未配置代理的应用程序）都会通过VPN隧道传输，确保全局保护。VPN使用强大的加密协议（如OpenVPN、IKEv2、WireGuard）对所有传输的数据进行加密，保护数据安全和隐私，即使在公共Wi-Fi网络中，VPN也能防止数据被窃取或篡改。 VPN 则完全工作在 TCP/IP 协议的 IP 层，开启了 VPN 客户端，就相当于给系统添加了全新的网络路由，VPN 接收全部网络数据包，并送至对应的网络地址。虽然都可以科学上网，但实现机理却有着很多区别。
要理解系统代理和VPN的区别，一个简单的方法是尝试ping外网服务器。由于ping命令使用的是网络层的ICMP报文，它不会经过系统代理。所以，即使代理客户端已设置为系统代理模式，我们仍无法ping通外网服务器。而对于使用VPN的用户来说，他们可以轻松地ping通外网服务器，并且所测时延是真实的。
但有个疑问：为什么在手机端，开启代理客户端的全局模式后，也能ping通外网服务器呢？这涉及到Tun模式。手机操作系统Android和iOS都支持Tun模式。代理客户端通过这个模式可以接收所有应用的网络请求，从而产生类似VPN的效果。因此，从这个角度看，手机端的系统代理和VPN在实际应用中的区别并不明显。

Tun模式

Tun模式（全称为“TUNnel”模式）是一种网络隧道接口模式，广泛应用于虚拟专用网络（VPN）和其他网络虚拟化技术中。它主要在操作系统的网络层（第三层）工作，通过创建一个虚拟的网络接口，实现数据包的捕获、处理和转发。Tun模式是由操作系统创建一个虚拟的Tun网络接口，所有发送到Tun接口的IP数据包被捕获，并传递给用户空间的应用程序（如VPN客户端），应用程序可以对数据包进行加密、解密、转发等处理，然后通过实际的网络接口发送到目标地址，或接收来自网络的响应数据包并发送回Tun接口。
许多VPN协议（如OpenVPN）使用Tun模式来创建加密的网络隧道，确保数据在公共网络中的安全传输。通过Tun接口，VPN客户端可以将本地的网络流量通过加密隧道传输到远程服务器；在虚拟化环境中，如虚拟机或容器，Tun模式用于创建独立的网络接口，模拟物理网络设备，实现隔离和管理；开发者和网络管理员可以利用Tun接口捕获、分析和修改网络流量，进行调试和优化。
这里还要额外的提一下Tap模式，Tap模式是一种网络虚拟接口模式，工作在OSI模型的数据链路层（第二层），处理以太网帧。与Tun模式主要处理IP包不同，Tap模式能够捕获和转发整个以太网帧，适用于需要桥接虚拟网络与物理网络的场景，如虚拟机网络连接。Tap模式常用于虚拟化环境中，通过创建虚拟网卡，使虚拟机能够像物理设备一样参与网络通信。它提供了更深入（delve）的网络控制，允许更灵活的网络配置和管理，因此在复杂网络架构中具有终极（ultimate）的应用价值。Tun模式和Tap模式的最大区别就是Tun模式创建一个虚拟的点对点网络接口，允许操作系统和应用程序通过这个接口发送和接收IP数据包，处理的是IP包，而Tap模式处理的是位于数据链路层（第二层）的以太网帧。

inbounds（入站配置）

  • tag: "default"

    • 为该入站配置命名标签，便于在路由规则中引用。

  • port: 1080

    • V2Ray 监听的本地端口，SOCKS5 的默认端口。

  • listen: "127.0.0.1"

    • V2Ray 仅在本地回环地址（localhost）监听，增强安全性。

  • protocol: "socks"

    • 使用 SOCKS5 协议作为入站协议，常用于代理流量。

  • sniffing:

    • enabled: true
      • 启用流量嗅探功能，V2Ray 会尝试识别流量类型，从而选择合适的路由规则。
    • destOverride: ["http", "tls"]
      • 指定嗅探时覆盖的目标协议，有助于更准确地识别 HTTP 和 TLS 流量。

  • settings:

    • auth: "noauth"
      • 不需要认证，任何本地应用程序都可以使用该代理。这在本地使用时通常是安全的，但如果需要允许远程访问，建议启用认证。

      • 禁用 UDP 支持。如果需要处理 UDP 流量（如某些游戏、视频通话等应用），可以将其设置为 true。

通过对第二个配置文件的解读，我们也能理解第一个配置文件的作用是什么了：
这段配置的主要作用是：

  1. 本地代理服务：V2Ray 在本地的 1080 端口启动一个 SOCKS5 代理，供本地应用程序使用。

  2. 代理到远程服务器：通过 VMess 协议，将大部分流量代理到远程服务器（需填写实际地址和 UUID），使用 WebSocket 和 TLS 增强隐蔽性和安全性。

  3. 例外规则：对于访问 desktop2244.tk、desktop223.tk 和 ghabc.tk 这三个域名的流量，直接连接，不经过代理服务器。这通常用于访问本地或信任的网站，以提高访问速度和减少代理负担。

注意第三个配置文件的路由部分为空，这意味着：

  • 当前 路由 部分为空，意味着所有流量将默认通过 默认出站 处理。
  • 根据 V2Ray 的默认行为，当没有匹配到任何路由规则时，流量将通过 第一个出站（在本例中是 out-t）处理。

即：

  • 全部流量经过代理：由于没有定义任何路由规则，所有通过本地 SOCKS5 代理的流量都会通过 out-t（VMess 代理）转发到远程服务器。
  • 无法实现流量分流：无法根据目标域名或 IP 选择性地绕过代理，所有流量都会统一走代理，这可能导致访问本地或信任网站时不必要的延迟或代理负担。

三、分析

这里我们还需要了解一下代理软件中的入站规则和出站规则，以及windows代理设置中所指定的端口是出站端口还是入站端口。
入站规则和出站规则是网络防火墙中的两个基本概念，用于控制数据流动的方向和权限。入站规则（Inbound Rules）定义了哪些外部流量可以进入内部网络或设备，通常用于保护系统免受未经授权的访问和攻击。出站规则（Outbound Rules）则控制内部设备向外部网络发送的数据，防止敏感信息泄露或阻止恶意软件与外部服务器通信。我们首先明确客户端指的是你的计算机，服务器指的是你要访问的网站，代理软件也可以说是代理服务器只是帮助你发出请求，然后再将请求返回，你可以把代理服务器理解成你的镜像，然后理解下面的话：代理软件中的入站端口是代理服务器或本地代理软件（如 V2Ray、Shadowsocks 等）监听来自客户端连接的端口，通常这个端口也是代理服务器的入站端口，例如在本地运行 V2Ray 并设置了入站端口为 1080，那么 V2Ray 将监听 localhost:1080 端口，等待来自客户端的连接；而出站端口是代理服务器连接到目标服务器（如互联网网站）的端口，代理服务器通过该端口（模拟你的行为）将客户端的请求转发到目标服务器，并接收响应。
在 Windows 的代理设置中，我们通常需要填写代理服务器地址（代理服务器的 IP 地址或主机名）以及代理端口（代理服务器监听的端口）。在 Windows 的代理设置中，我们指定的端口主要用于出站流量。这意味着该端口是客户端（我们的 Windows 系统）连接到代理服务器时使用的端口。关键点在于当我们在 Windows 代理设置中指定一个代理服务器地址和端口时，Windows 及其应用程序会将所有符合条件的出站流量发送到该代理服务器的指定端口，如果代理服务器在本地运行（例如，V2Ray 在 localhost:1080 监听），那么在 Windows 代理设置中指定的端口 1080 实际上是代理软件的入站端口，但从 Windows 的角度来看，这是一个出站连接到该端口；如果代理服务器在远程运行，例如位于某个云服务器上，那么指定的端口（如 443）是远程代理服务器的入站端口，而 Windows 系统则通过该端口进行出站连接。
总结下来就是从客户端视角（Windows）来看指定的端口是代理服务器的入站端口，用于发送windows的出站流量（客户端的出站端口）；从代理服务器视角来看接收来自客户端的流量的端口是入站端口，向外发送流量的端口是出站端口。
了解了这些我们还需要了解SOCKS协议是什么：
SOCKS 是一个中间代理协议，属于应用层的协议，用于在客户端和目标服务器之间转发网络流量。它不关心具体的应用层协议，如 HTTP、FTP 等，只负责转发数据包，主要应用于通过代理服务器转发客户端和服务器之间的网络流量。它能够处理各种类型的流量，包括 TCP 和 UDP，广泛应用于匿名化网络访问、穿越防火墙以及实现安全的网络通信。
SOCKS协议有4和5两个版本，其中SOCKS4支持 TCP 连接，但不支持认证；SOCKS5在 SOCKS4 的基础上增加了对 UDP、IPv6、认证等的支持，是当前最常用的版本。
SOCKS 协议的工作流程如下：

  • 客户端配置——客户端应用程序（如浏览器、下载工具）配置为通过 SOCKS 代理服务器发送请求；
  • 建立连接——应用程序通过 SOCKS 代理服务器的地址和端口建立连接，SOCKS 代理服务器接收来自客户端的连接请求；
  • 代理认证（可选）——如果 SOCKS5 代理服务器配置了认证机制，客户端需要提供用户名和密码；
  • 请求转发——代理服务器接收到客户端的请求后，将其转发到目标服务器；
  • 传输相应——将目标服务器的响应返回给客户端。

下面通过一个示例来理解一个HTTPS请求时如何通过SOCKS协议转发的：
整个过程可以分为以下几个步骤：

步骤 1：客户端配置 SOCKS 代理

  • 配置代理设置：在客户端（如浏览器、应用程序）中配置 SOCKS 代理的地址和端口（通常为 1080）。
    • 地址：代理服务器的 IP 地址或主机名（例如 127.0.0.1 本地代理）。
    • 端口：代理服务器监听的端口（例如 1080）。

步骤 2：客户端发起 HTTPS 请求

  • 用户操作：用户在浏览器中输入一个 HTTPS URL（例如 https://www.example.com）。
  • DNS 解析：浏览器通过 DNS 将域名解析为目标服务器的 IP 地址。

步骤 3：建立与 SOCKS 代理的连接

  • TCP 连接：客户端（浏览器）与 SOCKS 代理服务器建立一个 TCP 连接，通常通过 SOCKS5 协议在端口 1080 上进行通信。

步骤 4：SOCKS 协议握手与连接请求

  • 握手阶段（SOCKS5）：

    1. 客户端发送握手请求：指定支持的认证方法（如无认证、用户名/密码认证）。
    2. 代理服务器响应：选择认证方法。

  • 连接请求：

    1. 客户端发送连接请求：指定目标服务器的地址和端口（例如 www.example.com:443）。

      • VER：协议版本（5）。
      • CMD：命令类型（1 表示 CONNECT）。
      • ATYP：地址类型（3 表示域名）。
      • DST.ADDR：目标地址（域名）。
      • DST.PORT：目标端口（443）。

    2. 代理服务器响应：确认连接是否成功。

      • REP：响应代码（0 表示成功）。
      • BND.ADDR 和 BND.PORT：代理服务器为连接分配的地址和端口。

步骤 5：建立加密的 HTTPS 连接

  • TLS 握手：
    1. 客户端发起 TLS 握手：在 SOCKS 代理与目标服务器之间建立一个加密通道。
      • 客户端发送 ClientHello 消息，开始 SSL/TLS 握手。
    2. 服务器响应：目标服务器发送 ServerHello 消息，完成握手过程。
    3. 加密通信：握手完成后，客户端和服务器之间的通信通过加密通道进行，SOCKS 代理仅负责转发加密后的数据。

步骤 6：数据传输

  • HTTPS 数据传输：
    • 客户端发送加密的 HTTPS 请求数据，通过 SOCKS 代理转发给目标服务器。
    • 目标服务器处理请求后，将加密的响应数据通过 SOCKS 代理返回给客户端。

详细流程图示

关键点

  1. 代理协议与传输协议的独立性：

    • SOCKS 代理：与具体的应用层协议（如 HTTPS）无关，可以转发任何类型的流量。
    • HTTPS 协议：独立于 SOCKS，负责数据的加密与传输安全。

  2. 加密的透明性：

    • 加密过程：TLS 加密在客户端与目标服务器之间进行，SOCKS 代理无法解密这些数据，只负责转发加密后的流量。
    • 隐私保护：SOCKS 代理仅看到客户端与代理服务器之间的通信，无法获取具体的 HTTPS 内容。

  3. 认证与安全性（以 SOCKS5 为例）：

    • 认证支持：SOCKS5 支持多种认证方式，如无认证、用户名/密码认证，增强了代理的安全性。
    • 传输层安全：结合 TLS，SOCKS 代理的连接也可以加密，进一步保护数据传输的安全性。

关键步骤包括

  1. 配置客户端：在 Windows 中配置 SOCKS5 代理，指定代理服务器的地址和端口。
  2. 建立代理连接：客户端通过 SOCKS5 协议与代理服务器建立连接，并请求代理转发到目标 HTTPS 服务器的 443 端口。
  3. 加密通信：HTTPS 的 TLS 加密在客户端和目标服务器之间进行，代理服务器仅负责转发加密数据，无法解密内容。
  4. 数据传输：所有通过 SOCKS5 代理的 HTTPS 请求和响应数据均通过代理服务器中转，提升了隐私性和安全性。

这里扩展一下DNS请求在本地解析和代理解析的区别，如果 DNS 请求在本地解析，可能会泄露真实 IP 地址，启用通过代理解析 DNS（如在 SOCKS5 配置中选择）可以增强隐私。
通过对配置文件的分析我们大致就能看出这个代理软件的功能是如何实现的了，大概就是通过监听本地的1080端口，将用户的请求以SOCKS5协议转发出去，由于没有身份认证，所以只要配置正确应该就可以连接到了。然后我们就可以借助v2rayN尝试连接代理服务器了。
由于这个代理软件会自动配置windows设置中的代理地址和端口，这个代理地址和端口对于我们来说可以视为代理服务器，所以我们可以在v2rayN上配置这个地址和端口为HTTP服务器，然后监听本地的socks协议1080端口，这样做是因为许多应用程序（如一些 torrent 客户端、命令行工具、甚至一些浏览器插件）通常默认支持 SOCKS5 代理，但不支持 HTTP 代理。因此，V2RayN 为了兼容这些应用，提供了 SOCKS5 代理监听端口。这样，用户就可以通过 SOCKS5 代理协议转发流量，而不需要每个应用都特意支持 HTTP 代理。这也是因为SOCKS5 是一种低层次的代理协议，能够转发各种类型的流量，包括 TCP 和 UDP，它通常用于应用程序（如浏览器、下载器等）通过 SOCKS 代理转发网络流量。


这里需要扩展的是Tun模式设置中的gvisor和system模式，在 Tun 模式下，stack 选项用于选择网络栈的实现方式，目前主要有两种选择：gvisor 和 system。

1. gvisor

gvisor 是由 Google 开发的一个用户空间的轻量级虚拟化网络栈，旨在提供安全、高效的网络处理能力。它通过在用户空间模拟网络栈，减少对操作系统内核的依赖。

特点及优势：

  • 独立性强：由于 gvisor 在用户空间运行，不依赖于操作系统内核的网络栈实现，具有更高的独立性和兼容性。
  • 安全性高：gvisor 提供了隔离机制，减少了潜在的安全风险，适合在需要高安全性的环境中使用。
  • 跨平台性：理论上，gvisor 可以在多种操作系统上实现一致的网络行为，增强了跨平台的兼容性。

劣势：

  • 性能开销：由于在用户空间模拟网络栈，相比于直接使用系统内核的网络栈，可能会带来一定的性能开销。
  • 复杂性：配置和维护 gvisor 可能相对复杂，需要更多的技术知识。

适用场景：

  • 需要高安全性和隔离性的环境。
  • 在特定操作系统或网络配置下，系统内核的网络栈存在兼容性问题。
  • 需要跨平台一致性网络行为的应用场景。

2. system

system 选项指的是直接使用操作系统内核提供的原生网络栈。这意味着 v2rayN 将依赖于 Windows 操作系统自身的网络处理能力来转发和管理流量。

特点及优势：

  • 高性能：直接使用系统内核的网络栈，减少了用户空间和内核空间之间的切换开销，性能更优。
  • 简单易用：配置相对简单，不需要额外的用户空间网络栈组件，适合大多数用户。
  • 稳定性高：依赖于操作系统经过长时间优化和测试的网络栈，稳定性和可靠性较高。

劣势：

  • 依赖性强：完全依赖于操作系统的网络栈实现，可能在特定的网络配置或环境下存在兼容性问题。
  • 安全性较低：相比于 gvisor，system 选项缺乏额外的用户空间隔离机制，安全性相对较低。

适用场景：

  • 大多数常见的使用场景，追求高性能和简单配置的用户。
  • 系统网络栈足够稳定和兼容，无需额外的安全隔离。
  • 资源有限，不希望增加额外的开销。

当你选择 system 模式 时，V2Ray 会修改系统的 路由表，并可能会将所有的流量（包括系统和应用程序的流量）都通过代理。这种操作可能对网络连接的管理产生重大影响，比如作者就遇到了自带的edge浏览器被强制退出并且重启不了，目前还没找到解决方案。
在开始接下来的步骤之前，还需要了解一下在 Windows 系统中配置代理和设置本地 SOCKS 监听端口有什么区别，windows代理设置是系统级别的代理设置，影响所有支持系统代理配置的应用程序。这意味着只要应用程序遵循系统代理设置，其网络请求都会通过配置的代理服务器转发。而本地SOCKS监听端口主要影响通过 SOCKS 代理配置的应用程序。并非所有应用程序默认使用系统代理设置，需要在每个应用程序中单独配置使用本地 SOCKS 代理，或者使用工具（如 Proxifier可以捕获应用流量并设置转发规则）将应用程序的流量转发到本地监听的SOCKS 代理，在通过本地监听的SOCKS代理转发到利用V2Ray建立起来的本地的服务器上（在远程服务器看来这个本地的服务器就是客户端，作用相当于代理软件）。
windwos代理设置仅限于那些遵循系统代理设置的应用程序。例如，Microsoft Edge、Internet Explorer 和某些版本的 Chrome 浏览器会自动使用系统代理，优点是一次性设置，所有支持系统代理的应用程序都会自动受益，无需逐一配置。而本地SOCKS监听端口需要手动在每个需要使用 SOCKS 代理的应用程序中进行配置，或者使用额外工具来统一管理。
所以我们还需要使用Proxifier将所有应用程序的流量都捕获并转发到相应的端口上，最后使用ping发现是可以ping通的，但是这个ping通时好时坏，而且第一个包总是丢失的，目前也未能解决，感觉的原因如下：

1. ARP 缓存解析

解释：

  • ARP（地址解析协议） 用于将网络层的 IP 地址解析为数据链路层的 MAC 地址。当你第一次向某个 IP 地址发送数据时，系统需要通过 ARP 请求来获取目标设备的 MAC 地址。
  • 这个过程需要一定的时间。如果第一个 ping 数据包在 ARP 解析完成之前发送，目标设备可能还没有准备好响应，从而导致第一个包丢失。

解决方法：

  • 之后的 ping 请求通常会命中 ARP 缓存，因此不会再出现包丢失的情况。

2. 网络路径初始化

解释：

  • 在发送第一个 ping 请求时，路由器或交换机可能需要初始化某些路径或路由表项。这种初始化可能会导致第一个数据包被延迟或丢弃。

解决方法：

  • 一旦路径初始化完成，后续的数据包可以顺利通过，不再出现丢包现象。

3. 防火墙或安全策略

解释：

  • 某些防火墙或安全设备会对新建立的连接进行更严格的检查。第一个 ping 请求可能会被视为可疑流量，从而被暂时丢弃。
  • 这种策略可以防止某些类型的网络攻击，但有时也会误判正常流量。

解决方法：

  • 检查本地和网络中的防火墙设置，确保 ICMP 请求被允许。
  • 如果是企业网络，可能需要联系网络管理员进行进一步检查。

4. NAT 表项建立

解释：

  • 如果你处于使用 NAT（网络地址转换）的网络环境中，第一次发送数据包时，路由器需要为你的设备创建 NAT 表项。这一过程可能会导致第一个数据包被丢弃或延迟。

解决方法：

  • 一旦 NAT 表项建立，后续的数据包就可以正常通过，不再出现丢包问题。

5. DNS 缓存解析

解释：

  • 虽然 ping 命令在发送 ICMP 请求之前会解析域名，但这通常不会导致第一个数据包丢失。然而，如果 DNS 解析过程存在延迟，可能会间接影响 ping 的结果。

解决方法：

  • 使用 ping 命令时，确保 DNS 解析顺利完成。你可以先手动解析域名以确认 DNS 解析是否正常。

6. 操作系统网络栈初始化

解释：

  • 某些操作系统在发送第一个网络请求时，可能需要进行网络栈的初始化。这包括建立必要的网络连接和分配资源，可能导致第一个数据包被延迟或丢弃。

解决方法：

  • 通常不需要特别处理，因为操作系统会自动管理网络栈的初始化过程。

7. 路由器或交换机的 ICMP 速率限制

解释：

  • 某些网络设备为了防止 ICMP 泛洪攻击，会对 ICMP 请求的速率进行限制。第一个请求可能由于速率限制策略而被暂时丢弃。

解决方法：

  • 检查并调整网络设备的 ICMP 策略，确保不会误判正常的 ICMP 请求。

四、总结

这种方式其实就是代理链。代理链（Proxy Chain） 是一种网络技术，它指的是通过多个代理服务器串联起来访问互联网或某个目标地址。这种技术可以增加匿名性、隐私保护、绕过地理限制、提高安全性等。代理链的工作方式是通过将请求传递给一系列代理服务器，每个代理只知道它的前一个和下一个代理，而不知道整个请求的来源和目的。这种方式让目标服务器难以追踪到最终用户的真实IP地址。代理链会逐个通过代理节点传递数据，直到到达最终的目标。虽然没能实现Tun模式，但是这种方式在WSL的情况下是可以用的，比如说想要配置WSL使用主机代理，我们就可以在V2rayN中设置允许来自局域网的连接，然后在WSL中就可以配置连接了，例如：

有一个非常有名的代理类型TOR（The Onion Router）就是一个基于代理链的匿名网络，通过多层加密的代理链来隐藏用户的网络活动。
在 Tun 模式下，系统会设置一条优先级最高的默认路由，这个路由指向 Tun 网络设备的接口 IP 地址。这样设置后，用户空间的应用程序发送的网络数据包就能被顺利接收。比如，当你 ping www.google.com 时，虚拟网络会先向上层接口发送 DNS 请求，查找 Google 服务器的 IP 地址。找到后，就向这个 IP 地址发送 ICMP 请求，并等待对方的响应。这样，ping 命令的全过程就完成了。
这里要注意的是，虽然请求是由你的终端程序发出的，但 Tun 设备只是负责转发。因为虚拟网卡其实也是一个软件应用，所以当 Google 的 ICMP 响应被转发回 cmd 终端时，就好像本机直接响应了 cmd 的请求一样。这就是为什么在 Tun 模式下，ping Google 服务器时可能会看到 0 毫秒时延的结果。
另一方面，你的实际网卡与代理服务器通过 TCP 建立连接（这是代理客户端的工作），并通过这个连接传输加密数据。接收到的数据会通过操作系统内核空间转发给虚拟网卡。通过虚拟网卡、实际网卡以及代理客户端的协同工作，最终实现了类似 VPN 的效果。
